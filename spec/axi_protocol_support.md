# AXI protocol support

## Implemented interface profile

The controller implements a deliberately constrained AXI4-compatible slave profile:

- 32-bit address and 32-bit data channels.
- One write response per accepted AW/W pair.
- One read response per accepted AR request.
- AW and W may arrive independently. W is held by the master until an AW entry is available.
- Up to eight queued AW requests and eight queued AR requests at the AXI clock boundary.
- Responses are returned in acceptance order within each channel.
- B and R payloads remain stable while VALID is asserted and READY is low.
- Reset cancels all queued and active AXI transactions and clears BVALID/RVALID.
- No AXI ID ports are implemented; therefore transaction reordering by ID is not supported.

## Supported transfer encoding

Production software and bus fabric shall generate:

- `AxLEN = 0` (single beat).
- `AxSIZE = 2` (four bytes per beat).
- `AxBURST = INCR` (`2'b01`).
- Word-aligned addresses (`AxADDR[1:0] = 0`).
- `WLAST = 1` for the accepted write data beat.

The current controller data path executes one memory operation for every accepted request. Burst expansion, FIXED/WRAP address generation, narrow transfers, and unaligned accesses are not implemented.

## Deterministic handling of unsupported encodings

Until a dedicated front-end error responder is introduced, unsupported `AxLEN`, `AxSIZE`, `AxBURST`, unaligned address, or `WLAST` encodings are consumed as one controller operation and complete with the normal single response. They must not be generated by conforming integration logic. M6 assertions and CI tests identify these encodings at the verification boundary so their use cannot be mistaken for supported AXI behavior.

This policy is intentional and deterministic, but it is not full AXI4 burst support. A future interface revision may replace this compatibility behavior with `SLVERR` without changing the supported profile above.

## Verification requirements

M6 CI verifies:

- AW-before-W and W-before-AW channel arrival.
- Multiple queued writes and reads.
- B/R backpressure stability.
- Response ordering and `RLAST` behavior.
- Reset while AW, W, B, AR, or R activity is pending.
- Detection of unsupported request encodings.

The `agent/m6-ci-validation` branch is used only to force a complete pull-request workflow validation of the M6 implementation before fast-forwarding the validated revision to `main`.
